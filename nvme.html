NVMe Backup & Storage Plan

— mounting the new 1 TB NVMe at /mnt/storage is a good plan. I would structure it as dedicated backup + shared storage, with controlled group permissions.

Important note: because the main SSD is 2 TB and the backup SSD is 1 TB, do not plan this as a full disk clone unless the used data is well below 1 TB. Use it for selected backups: /home, important configs, Docker volumes, project folders, databases, Mattermost data, etc.

Recommended layout

/mnt/storage/
├──────────── shared/ # files shared by selected users
├──────────── backups/ # automated system/application backups
│ ├────────────────── home/
│ ├────────────────── etc/
│ ├────────────────── docker/
│ ├────────────────── databases/
│ └────────────────── logs/
└──────────── staging/ # temporary copy/import area

Use a Linux group such as:

backupgroup

Then add the two trusted users to this group.

IMPORTANT: Make a note for the steps you have done

1. Identify the new NVMe safely

After installing the SSD, boot Linux and run:

# List all block devices with columns for name, size, model, serial, filesystem type, and mount points
lsblk -o NAME,SIZE,MODEL,SERIAL,FSTYPE,MOUNTPOINTS

Also useful:

# List all connected NVMe solid state drives and their specific namespaces
sudo nvme list

You may see something like:

nvme0n1   2T   existing system disk
nvme1n1   1T   new storage disk

Be very careful: the device name may be different. In the commands below, I will assume the new SSD is:

/dev/nvme1n1

Do not run formatting commands until you are 100% sure.

2. Partition and format the new SSD

For simple, reliable Linux backup storage, I suggest ext4 !!!!!Be 100% sure you are putting the new SSD name not the existing SSD name in the steps to proceed each line.

!!!!!STOP IF YOU ARE NOT SURE!!!!!.

# Create a new GPT partition table on the specified NVMe drive
sudo parted /dev/nvme1n1 --script mklabel gpt
# Create a single primary partition spanning 100% of the available drive space
sudo parted /dev/nvme1n1 --script mkpart primary ext4 0% 100%
# Format the newly created partition with the ext4 filesystem and assign it the label "storage1tb"
sudo mkfs.ext4 -L storage1tb /dev/nvme1n1p1

Check it:

# List block devices along with their active filesystem types and UUIDs
lsblk -f

You should see something like:

nvme1n1p1  ext4  storage1tb  <UUID>

3. Mount it at /mnt/storage

Create the mount point:

# Create the mount point directory, creating parent directories if necessary (-p)
sudo mkdir -p /mnt/storage

Get the UUID:

# Retrieve the UUID and filesystem attributes of the new partition
sudo blkid /dev/nvme1n1p1

Example output:

/dev/nvme1n1p1: LABEL="storage1tb" UUID="abcd-1234-..." TYPE="ext4"

Edit /etc/fstab:

# Open the filesystem table file in the vim text editor to configure persistent mounting
sudo vim /etc/fstab

Add this line, replacing the UUID with your actual UUID:

UUID=abcd-1234-xxxx /mnt/storage ext4 defaults,noatime,nodev,nosuid 0 2

Then test:

# Mount all filesystems specified in /etc/fstab to verify the new configuration is valid
sudo mount -a
# Display disk space usage for the new mount point in human-readable format (-h)
df -h /mnt/storage

If no error appears and /mnt/storage shows about 900+ GB usable space, the mount is good.

4. Create the group and add users

Create the group:

# Create a new system group named "backupgroup"
sudo groupadd backupgroup

To identify the exact usernames of the regular users on your system before adding them, you can list them using either of these commands:

# Option A: List the home directories (usually corresponds to actual human user accounts)
ls -1 /home

# Option B: Filter the passwd file for regular users (typically with a User ID of 1000 or greater)
awk -F: '$3 >= 1000 && $3 != 65534 {print $1}' /etc/passwd

Add the two users to the group. Replace user1 and user2 with the real Linux usernames you identified above:

# Append (-a) user1 to the newly created backupgroup (-G)
sudo usermod -aG backupgroup user1
# Append (-a) user2 to the newly created backupgroup (-G)
sudo usermod -aG backupgroup user2

Example:

# Example: Add the user 'andrew' to the backupgroup
sudo usermod -aG backupgroup andrew
# Example: Add the user 'tiger' to the backupgroup
sudo usermod -aG backupgroup tiger

Normally, a user must log out (terminate the SSH session) and log back in for group changes to take effect. If you want to apply the new group to your current terminal session immediately without logging out, you can switch user to yourself:

# Example: Start a new login shell as your user 'andrew'(if you logged in as 'andrew') to immediately apply the new group to the current session
su - tiger

# Alternatively, you can use newgrp to temporarily log into the new group in the current shell
newgrp backupgroup

Check:

# Verify the group memberships for the user 'andrew'
groups andrew
# Verify the group memberships for the user 'tiger'
groups tiger

# Verify the active group memberships for the current session
groups

5. Create folders with proper permissions

Create the directory structure:

# Create the base directory structure for shared files, backups, and staging
sudo mkdir -p /mnt/storage/shared
sudo mkdir -p /mnt/storage/backups/home
sudo mkdir -p /mnt/storage/backups/etc
sudo mkdir -p /mnt/storage/backups/docker
sudo mkdir -p /mnt/storage/backups/databases
sudo mkdir -p /mnt/storage/backups/logs
sudo mkdir -p /mnt/storage/staging

Set ownership:

# Recursively change the owner to root and the group to "backupgroup" for all files and directories in storage
sudo chown -R root:backupgroup /mnt/storage

Set permissions:

# Set permissions to rwxrws--- (owner and group full access, SetGID bit set to force group inheritance)
sudo chmod 2770 /mnt/storage/shared
sudo chmod 2770 /mnt/storage/staging
sudo chmod 2770 /mnt/storage/backups
sudo chmod 2770 /mnt/storage/backups/*

The 2 in 2770 is important. It means new files created inside these folders inherit the group backupgroup.

Optional but recommended: use default ACLs so new files are group-writable.
Note: If the setfacl command is not found, install it via sudo apt install acl (Debian/Ubuntu) or sudo dnf install acl (RHEL/CentOS).

# Set Access Control Lists (ACLs) to ensure existing files grant read/write/execute permissions to backupgroup
sudo setfacl -R -m g:backupgroup:rwx /mnt/storage/shared
# Set default ACLs to ensure newly created files inherit group read/write/execute permissions
sudo setfacl -R -m d:g:backupgroup:rwx /mnt/storage/shared

# Apply the same standard ACLs to the staging directory
sudo setfacl -R -m g:backupgroup:rwx /mnt/storage/staging
# Apply the same default ACLs to the staging directory
sudo setfacl -R -m d:g:backupgroup:rwx /mnt/storage/staging

For backups, I would be more careful. Usually only root or a backup service should write there. Users can read if needed, but not casually modify backup history.

A safer backup permission model:

# Ensure the backups folder is owned by root and the backupgroup
sudo chown -R root:backupgroup /mnt/storage/backups
# Set permissions so root has full access, the group can read/enter, and others have no access (2750)
sudo chmod -R 2750 /mnt/storage/backups

That means:

  • root: full control
  • backupgroup: read/enter
  • others: no access

For shared files:

# Apply full group access and SetGID to the shared directory
sudo chmod 2770 /mnt/storage/shared

That means:

  • owner/root: full control
  • backupgroup: full control
  • others: no access

6. Recommended backup approach

For simple local backups, start with rsync.

Create a backup script:

# Create and open a new shell script file for the backup routine using vim
sudo vim /usr/local/sbin/local-backup.sh

Paste:

#!/bin/bash
# Exit immediately if a command exits with a non-zero status, treat unset variables as an error, and catch pipe failures
set -euo pipefail

DEST="/mnt/storage/backups"
DATE="$(date +%F_%H-%M-%S)"
LOG="$DEST/logs/backup-$DATE.log"

# Create required backup subdirectories if they don't exist
mkdir -p "$DEST/home" "$DEST/etc" "$DEST/docker" "$DEST/logs"

# Log the exact start time of the script to the log file
echo "Backup started at $(date)" | tee -a "$LOG"

# Sync home directory, preserving permissions/attributes, and excluding cache and downloads directories
rsync -aAXHv --delete \
  --exclude='.cache/' \
  --exclude='Downloads/' \
  /home/ "$DEST/home/current/" | tee -a "$LOG"

# Sync system configuration files from /etc
rsync -aAXHv /etc/ "$DEST/etc/current/" | tee -a "$LOG"

# Export docker container names and volume lists if the docker command is available
if command -v docker >/dev/null 2>&1; then
  docker ps --format '{{.Names}}' > "$DEST/docker/running-containers-$DATE.txt" || true
  docker volume ls > "$DEST/docker/docker-volumes-$DATE.txt" || true
fi

# Log the completion time of the script
echo "Backup finished at $(date)" | tee -a "$LOG"

Make it executable:

# Make the newly created backup script executable
sudo chmod +x /usr/local/sbin/local-backup.sh

Test it manually:

# Execute the backup script manually to verify it runs without errors
sudo /usr/local/sbin/local-backup.sh

Check:

# List the contents of the backups directory to verify the backup completed successfully
ls -lah /mnt/storage/backups

7. Schedule automatic backup

Use cron:

# Edit the cron table for the root user to schedule automated tasks
sudo crontab -e

Add this for daily backup at 2:30 AM:

# Cron job syntax: run the local-backup.sh script every day at exactly 2:30 AM
30 2 * * * /usr/local/sbin/local-backup.sh

Check logs later:

# List the log files to ensure scheduled backups are running properly
ls -lah /mnt/storage/backups/logs

8. Better backup option: Restic or Borg

For a more professional backup system, I would use restic or borgbackup instead of plain rsync.

Why?

Plain rsync gives you a mirror. If a file is deleted or corrupted and rsync runs, the backup can also lose that file.

Restic/Borg gives you:

  • snapshots
  • deduplication
  • retention policy
  • compression
  • optional encryption
  • easy restore by date

A good design would be:

/mnt/storage/restic-repo

Then backup:

  • /home
  • /etc
  • /opt
  • /srv
  • important Docker volumes
  • Mattermost/PostgreSQL dumps

For a clinic/server environment, I would prefer restic or borg long-term.

9. Docker / database backup warning

If this Linux box will run things like Mattermost, PostgreSQL, MariaDB, OSCAR, or other Docker services, do not only copy raw database files while the database is running.

Better:

  • PostgreSQL → pg_dump
  • MariaDB/MySQL → mysqldump
  • Docker volumes → backup after dump, or stop service briefly

To find the exact name of your database container before running the backup command, list your active containers:

# Display a clean table of all running containers with their names and images
docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Status}}"

Example PostgreSQL dump style (replace postgres-container with the actual name you found above):

# Execute a full database dump from within the running PostgreSQL container and save it to the backups folder
docker exec postgres-container pg_dumpall -U postgres > /mnt/storage/backups/databases/postgres-$(date +%F).sql

Mattermost usually uses PostgreSQL, so you should back up both:

  • Mattermost config/data files
  • PostgreSQL database dump
  • uploaded files directory
  • Docker compose file
  • .env file

10. My recommended final plan

Use this setup:

  • Disk mount: /mnt/storage
  • Filesystem: ext4
  • Group: backupgroup
  • Shared writable folder: /mnt/storage/shared – owner root:backupgroup, permission 2770
  • Backup folder: /mnt/storage/backups – owner root:backupgroup, permission 2750
  • Backup method: start with rsync, later upgrade to restic or borg
  • Schedule: daily backup at 2:30 AM
  • Backup targets: /home, /etc, /opt, /srv, Docker compose files, database dumps, important project folders
  • Do not: treat the 1 TB disk as a full mirror of the 2 TB system SSD

For your case, I would use:

/mnt/storage/shared

for human-accessible shared storage, and:

/mnt/storage/backups

for automated backups only. That separation is cleaner and safer.
